\section{Methodology}

\subsection{Research Hypothesis and Statistical Analysis}

The central hypothesis of this study is formulated as follows:

\begin{equation*}
H_{0}:
\begin{minipage}[t]{0.8\linewidth}
\gls{SMU} (PM-table) and \gls{RAPL} measurements for the \emph{package} and
\emph{core} domains do not exhibit systematic differences and demonstrate a
high degree of dynamic correlation.
\end{minipage}
\end{equation*}

To evaluate this hypothesis, a combination of complementary statistical methods
was applied:
\begin{itemize}
    \item the Wilcoxon signed-rank test to detect potential systematic bias,
    \item Pearson's correlation coefficient and cross-correlation analysis
        to assess the strength of the dynamic relationship and to identify
        possible time lags,
    \item Bland–Altman analysis, complemented by numerical error metrics
        such as \gls{RMSE}, \gls{MAE}, and \gls{MAPE}, to quantify dispersion
        and define limits of agreement.
\end{itemize}

This combination of tests was selected to capture different aspects of agreement
between the two measurement sources: Wilcoxon provides sensitivity to systematic
shifts in central tendency, correlation and cross-correlation focus on temporal
coherence and dynamic similarity, while Bland–Altman and error metrics assess
the absolute level of agreement and practical interchangeability.  

All measurement series were re-sampled to a uniform time step of 100–200~ms.  
In cases where a lag was detected between the compared signals, one series was
shifted accordingly prior to performing the statistical tests.

\subsection{Test System Setup}

All experimental data in this study was collected on a system equipped with an
AMD Ryzen 5 3600 processor, featuring six physical cores and twelve hardware
threads. The processor operated on a Gigabyte A320M-H-CF mainboard, running
\gls{BIOS} version F53 (release date: January 2021).  

For reproducibility, the complete hardware and software specifications of the
test environment are provided in \cref{app:hwspec}. This includes detailed
\gls{CPU}, memory, storage, \gls{GPU}, and firmware information, as well as
the software stack and driver versions used during the experiments.

\subsection{Supporting Documentation}

In addition to research articles, several forms of supporting
documentation were instrumental in the preparation and execution of
this study. These resources provided both the practical foundation for
system configuration and the technical background required for working
with AMD-specific interfaces, Linux monitoring tools and kernel drivers.

The Red Hat documentation and ArchWiki were frequently consulted during
system setup and software preparation. These resources were essential in
the identification and configuration of key tools, including hardware
monitoring utilities, processor and/or mainboard topology inspection
tools, kernel modules and drivers, process management mechanisms, etc.
Beyond initial configuration, both sources served as references for
troubleshooting and verification of software and hardware functionality for
target system, thereby contributing directly to the reliability of the
experimental environment and setup.
\parencite{RedHat_Docs, ArchWiki}

For AMD-specific mechanisms, official vendor documentation proved less
useful due to limited accessibility and fragmented nature. In practice,
one the most valuable reference was the open-source \texttt{ryzen\_smu} kernel
driver repository \parencite{RyzenSMU_GitHub}. Alongside with the source code
it contains fairly detailed documentation in its accompanying README.
To overcome some challenges during methodology development the
\texttt{ryzen\_monitor} repository \footcite{RyzenMonitor_GitHub} was also used
as a source of some implementation and usage examples of the driver. Direct
inspection of the mentioned material provided critical insights into available
usage variants, functions, register mappings, and constraints, which were essential
for the implementation of the required/missing software parts required for
successful completions of this research.

Intel’s official developer manuals, which are widely regarded
as a reference standard in terms of accessibility and comprehensiveness,
were used primarily as a comparative benchmark \parencite{Intel_SDM}. Their
structured presentation of hardware interfaces and architectural features
highlighted the relative scarcity and fragmentation of AMD-specific documentation.

A complementary source of technical insight is the informal GitHub
repository \\\emph{plops/ryzen\_managment\_linux}, which provides annotated
explanations of the low-level behavior of the \texttt{ryzen\_smu} driver
interface. In particular, the author highlights that the SMU driver is
\emph{on-demand} rather than polling, and implements an internal throttling
mechanism—limiting \gls{SMU} refreshes to at most 1 kHz—even if userspace attempts
to poll faster \parencite{ryzen_managment_linux}. While this project is not a
formal documentation, it helps to understand practical limitations of
the \texttt{pm\_table} interface and guides decisions about sampling
frequency and overhead optimization during plugin implementation.

\subsubsection{Official Telemetry Sources}

This study relies on a combination of vendor documentation, official
tools, and in-band libraries to establish a reproducible and verifiable
telemetry baseline for AMD hardware. These sources provide the
architectural context, recommended system tuning, and \gls{API}/driver-level
details required to implement and validate measurement adapters.

\begin{itemize}
  \item \textbf{AMD architecture and power whitepapers.} Official AMD
  whitepapers describe the System Management Unit (SMU), per-die agents,
  and power domains on EPYC/Ryzen platforms. These documents are used to
  define measurement scope (which hardware domains are exposed), to
  interpret telemetry semantics, and to justify the comparability
  between SMU-derived and RAPL-derived metrics
  \parencite{AMD_EPYC_9004_8004_PowerMgmt_2024,
  AMD_EPYC_5th_Arch_2024}.

  \item \textbf{In-band libraries and E-SMI.} Libraries such as E-SMI
  (esmi\_ib\_library) expose programmatic access to SMU/EPYC telemetry.
  These \gls{API}s were consulted for implementation patterns and (where
  applicable) used to validate readings obtained through the
  \texttt{ryzen\_smu} driver \parencite{AMD_esmi_ib_library}.

  \item \textbf{Tuning and platform configuration guides.} Official
    platform tuning guides and vendor recommendations (OS tuning, \gls{CPU}
  governors, BIOS settings) were followed to ensure reproducible test
  conditions and to avoid systemic distortions of power measurements
  \parencite{AMD_RHEL_Tuning_9004_2024}.

  \item \textbf{Security and limitations.} Research on power-reporting
  side channels motivates conservative interpretation of high-resolution
  telemetry and informs safety/privacy mitigations when exposing raw
  power or energy counters \parencite{PwrLeak_2023}.
\end{itemize}

The combined use of these official documents and tools supports: (i)
correct identification of measurement domains; (ii) reproducible
platform configuration; (iii) selection of implementation approaches fo
the \gls{SMU}-to-energy adapter; and (iv) principled discussion of the limitations
and security considerations of telemetry sources.

\subsection{Limitations and Considerations}
\label{sec:limitations}

Several limitations should be acknowledged regarding the measurement setup
and methodology.

First, the \gls{RAPL} interface on Intel-compatible CPUs enforces a hardware
constraint on the update frequency of energy counters. Independent of how
often the counters are read, their effective resolution is limited to about
1~kHz. This design choice has been discussed in the context of limiting the
risk of side-channel attacks exploiting high-resolution power traces
\cite{intel_rapl,wang2023power}.

In contrast, preliminary experiments with the Ryzen \gls{SMU} driver suggest
substantially higher sampling capabilities. Power values from the PM-table can
be polled at microsecond-scale intervals without immediately saturating, thus
offering the potential for higher temporal resolution than RAPL. While this
feature was not systematically explored in the present study, it indicates a
promising direction for future research. A minimal reference implementation
used for polling-rate validation is available in the supplementary repository
\cite{yahdzhyiev2025repo}.

Second, it must be emphasized that the Ryzen \gls{SMU} driver employed in this
work is not an official AMD release. Its functionality is largely the result
of community-driven reverse engineering efforts
\cite{ryzen_monitor,plops2025ryzen}. Consequently, exact semantics of some
registers and tables remain partially undocumented, which may introduce
uncertainties in the interpretation of results. Nevertheless, such community
tools represent the current state of the art for accessing low-level telemetry
on AMD platforms in the absence of official vendor support.

\subsection{Energy Measurement Framework}

For this study, the \gls{EMA} framework was selected as the primary tool
for collecting energy consumption data. The choice was motivated by several
key advantages:

\begin{itemize}
    \item \textbf{In-code integration:} \gls{EMA} allows precise measurement of
          energy consumption for specific regions of application code,
          enabling multiple benchmarks to be integrated within a single code base.
    \item \textbf{Plugin-based extensibility:} The framework supports multiple
          hardware measurement back-ends (e.g., \gls{RAPL}, Ryzen-specific sensors)
          concurrently through a uniform public interface, without modifying
          the application source.
      \item \textbf{Reliability:} \gls{EMA} handles common measurement pitfalls,
          such as counter overflows in \gls{MSR}s, ensuring accurate and
          consistent data.
    \item \textbf{Familiarity and openness:} Authors have prior experience with
          \gls{EMA} and contributed to its development, facilitating integration
          and future tool improvements.
\end{itemize}

\gls{EMA}'s architecture is structured into layers, as illustrated in
\cref{fig:ema_arch}. The core of the framework is the \emph{Plugin} system,
which encapsulates device-specific measurement logic while exposing a
consistent interface to the user. The \emph{User Layer} provides both
high-level and low-level \gls{API}s: the high-level \gls{API} offers macros
for defining measurement regions directly in code, whereas the low-level
\gls{API} allows fine-grained control over plugin management, region
lifecycles, and data output.\parencite{NAAICE}

These design choices make \gls{EMA} well-suited for integrating energy
measurements into experimental workloads and benchmarks, supporting
reproducibility and direct comparison across different measurement backends
\parencite{NAAICE}.
